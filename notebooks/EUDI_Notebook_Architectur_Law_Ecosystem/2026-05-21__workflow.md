# EUDI Notebook Workflow

Date: 2026-05-21
Status: Active workflow
Notebook: EUDI Notebook — Architecture, Law & Ecosystem

---

## Purpose

This workflow defines how research notes should be created, updated, reviewed, and connected to the project.

The goal is to avoid collecting disconnected information. Every serious note should explain what the source says, what it means, why it matters, and what action follows from it.

---

## Core note workflow

Use this structure for every serious EUDI research note:

```text
Source → Meaning → Relevance to my project → Open questions → Next action
```

This structure keeps research useful for the prototype.

---

## Section explanation

### Source

Record where the information came from.

Include:

- source title
- organization or author
- link if available
- document section if known
- access date if useful
- source type

Example source types:

```text
Official EU source
Official GitHub repository
Architecture and Reference Framework
Regulation / legal text
Technical specification
Pilot / ecosystem source
Secondary explanation
```

Prefer official sources first.

---

### Meaning

Explain the source in plain language.

This section should answer:

- What does this source say?
- What is the concept?
- Which actors are involved?
- What problem does it solve?
- What responsibility or rule does it describe?

Use your own words. Do not copy large sections from the source.

---

### Relevance to my project

Explain how the source affects the prototype.

This section should answer:

- Does this affect the relying-party side?
- Does this influence verification requests?
- Does this influence wallet response handling?
- Does this influence audit logs?
- Does this influence business process modeling?
- Does this create a requirement?
- Does this define something as out of scope?

Use this section to connect research to project decisions.

---

### Open questions

Record what is still unclear.

Good open questions are specific.

Weak example:

```text
How does EUDI work?
```

Better example:

```text
What exact information must a relying party provide before requesting PID attributes from a wallet user?
```

Open questions should later move into:

```text
2026-05-21__open_questions.md
```

---

### Next action

Define what should happen next.

Examples:

```text
- Add definition to glossary.
- Compare this with ARF terminology.
- Turn this into a requirement candidate.
- Add this concept to the business relevance matrix.
- Mark this as out of scope for version 1.
- Research the official verifier endpoint repository.
```

Every serious note should end with at least one concrete next action.

## Research-to-project process

Use this process when reading a source:

1. Save the source in the source register.
2. Create or update a source note.
3. Extract important terms into the glossary.
4. Convert unclear points into open questions.
5. Convert project-relevant points into concept notes or requirements.
6. Add important decisions to project relevance notes.
7. Summarize progress in the weekly review.

---

## When to create a new file

Create a new file when:

- the topic is large enough to stand alone
- the note is based on a major source
- the note may become useful for documentation
- the note creates a project decision
- the note explains a reusable EUDI concept

Do not create a new file for every small thought. Small thoughts belong in open questions or an existing note.

---

## When to update an existing file

Update an existing file when:

- the concept is the same
- you are improving clarity
- you are adding a source to the same topic
- you are fixing mistakes
- the note is still part of the same research step

If the meaning changes substantially, update the date in the filename or create a new dated update file.

---

## Date handling

The date in the filename represents the latest meaningful content version.

Use a new date when:

- the note receives major new content
- your understanding changes
- a new source changes the conclusion
- the note becomes a new version

Do not use a new date for:

- typo fixes
- formatting cleanup
- small wording improvements

---

## Status labels

Use one of these labels near the top of each note:

```text
Status: Draft
Status: In review
Status: Stable
Status: Needs update
Status: Superseded
```

Meaning:

```text
Draft        = incomplete first version
In review    = mostly written but still needs checking
Stable       = good enough to use as project reference
Needs update = likely outdated or incomplete
Superseded   = replaced by a newer note
```

## Tag system

Use tags when helpful:

```text
[SOURCE]     Official or important source
[CONCEPT]    Important EUDI concept
[LEGAL]      Legal or regulatory point
[ARCH]       Architecture point
[TRUST]      Trust model point
[PRIVACY]    Privacy or data minimization point
[PROJECT]    Relevant to this prototype
[DECISION]   Project decision
[QUESTION]   Open question
[IHK]        Useful for IHK/project documentation
```

Keep tags simple. They should help scanning, not replace clear writing.

---

## Quality checklist

Before considering a note useful, check:

- [ ] Is the source clear?
- [ ] Is the meaning written in plain language?
- [ ] Is the project relevance explicit?
- [ ] Are open questions specific?
- [ ] Is there a next action?
- [ ] Are terms added to the glossary if needed?
- [ ] Are project-relevant points moved to the project relevance folder?

---

## Main rule

Do not collect information only because it is interesting.

Collect information because it helps explain, scope, justify, or improve the EUDI Wallet prototype.
