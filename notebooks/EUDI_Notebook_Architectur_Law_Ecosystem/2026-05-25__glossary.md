## ARF Definitions — simple explanation and examples

| Term | Simple explanation | Simple example | Relevance to my project |
|---|---|---|---|
| Attribute | A single piece of information about a person, organization, or entity. | Date of birth, age over 18, address, driving licence category, company registration number. | Verification templates request attributes. The project must help decide which attributes are necessary and which are excessive. |
| Authentic Source | The original or legally recognized source of truth for an attribute. | A population register for legal name/address, a driving licence authority for driving privileges, a company register for company data. | The prototype should distinguish between user-provided data and data that comes from a trusted/authentic source. |
| Electronic Attestation of Attributes (EAA) | A digital proof that confirms certain attributes. | A digital proof that someone has a university degree, professional qualification, or residence status. | The project may simulate wallet responses that contain attribute attestations instead of full identity documents. |
| Issuer | The actor that gives PID or attribute attestations to the user/wallet. | A state authority issues PID; a university issues a degree attestation; a professional chamber issues a licence attestation. | Simulated wallet responses should show who supposedly issued the data, because issuer trust affects whether the relying party accepts it. |
| National Accreditation Body (NAB) | A national body that accredits organizations that assess conformity. | A national accreditation body authorizes a conformity assessment body to assess whether a wallet solution meets required standards. | Mostly background for version 1. It helps explain certification and trust governance, but the prototype does not perform accreditation. |
| Person Identification Data (PID) | Core identity data used to establish who a natural or legal person is. | Name, date of birth, personal identifier, nationality, or legal-person identity data depending on context. | PID is central when a business process needs identity proof. The prototype must avoid requesting full PID when a smaller proof is enough. |
| Person Identification Data Provider | A Member State or legal entity that provides PID to users. | A national identity authority provides identity data to a citizen's wallet. | In version 1 this is simulated, but the project should still model PID as coming from a responsible provider. |
| Public Key Infrastructure (PKI) | A technical trust system for managing public keys and checking that digital signatures belong to the right entity. | A relying party checks a credential signature using trusted public keys. | Full PKI validation is out of scope for version 1, but the audit log can record simulated trust checks. |
| Qualified Electronic Attestation of Attributes Provider | A qualified provider that issues high-trust electronic attestations of attributes. | A qualified provider issues an official professional qualification attestation. | Useful for future high-assurance scenarios. In version 1, QEAA can be represented as a higher-trust credential type. |
| Qualified Electronic Signature Creation Device (QSCD) | Approved software or hardware used to create qualified electronic signatures. | A secure signing device or certified signing app used to create a qualified electronic signature. | Mostly out of scope for the current verifier prototype unless future use cases include signing. |
| Qualified Trust Service Provider (QTSP) | A trust service provider that has qualified status from the supervisory body. | A qualified provider that issues qualified certificates or qualified attestations. | Important trust ecosystem context. The prototype should not claim to be a QTSP. |
| Relying Party | A person or organization that relies on electronic identification or trust services. In the wallet context, it relies on data presented from an EUDI Wallet. | A bank, employer, public office, online platform, or age-restricted service requesting wallet proof. | This is the main project perspective. The prototype supports the relying-party side by turning business needs into verification requests. |
| Selective Disclosure | The wallet capability that lets a user share only selected attributes instead of everything. | Sharing only proof of being over 18 instead of full date of birth and address. | Core privacy principle for the prototype. The system should warn when a request asks for more data than needed. |
| Trust | Willingness to rely on another actor to perform actions or make claims. | A relying party trusts that a PID provider issued correct identity data. | The prototype should make trust assumptions explicit, especially when simulated data is accepted. |
| Trust Framework | A legally enforceable set of operational and technical rules for a multi-party system. | Rules that define who may issue credentials, how they are checked, and what obligations participants have. | Useful for explaining that EUDI is not just technology; it depends on governance, rules, and responsibilities. |
| Trust Model | The rules that ensure ecosystem components and actors are legitimate. | Checking that a wallet, issuer, credential type, and relying party are recognized and valid. | Version 1 can simulate trust checks as business logic, while clearly stating that real trust validation is out of scope. |
| Trust Service Provider (TSP) | A person or organization that provides one or more trust services, qualified or non-qualified. | A provider offering electronic signatures, seals, timestamps, or attribute attestations. | Helps separate normal service providers from qualified providers. Mostly background unless a use case requires trust services. |
| Trust Service | A regulated electronic service connected to signatures, seals, timestamps, attestations, certificates, archiving, remote signing devices, or ledgers. | Electronic signature validation, timestamping, electronic attestation of attributes, or website authentication certificates. | Helps understand what kind of services exist around identity and trust. The prototype should not pretend to provide regulated trust services. |
| Trusted List | A list of authoritative entities and their current or historical trust status. | A list showing which providers are trusted or qualified in a legal context. | Real trusted-list integration is out of scope for version 1, but the concept is important for future validation logic. |
| User | The natural or legal person using an EUDI Wallet. | A citizen using a wallet to prove identity, or a company representative using a wallet for business authentication. | The prototype must respect that the user controls what is presented from the wallet. |
| EUDI Wallet Instance | A specific wallet installation or instance that belongs to and is controlled by a user. | The wallet app installed and activated on one person's phone. | Useful for understanding that the user interacts through a concrete wallet instance, not an abstract system. |
| EUDI Wallet Provider | The public or private organization responsible for operating an eIDAS-compliant wallet solution. | A Member State or approved organization providing a wallet app. | Out of scope for building, but important for ecosystem mapping. The project is not a wallet provider. |
| EUDI Wallet Solution | The complete wallet product and service offered by a wallet provider. | A certified wallet app plus supporting backend/service environment. | Helps define project boundaries: this prototype is not an EUDI Wallet Solution. |

## Most important terms for this prototype

The most important terms for the current version are:

```text
Relying Party
Attribute
PID
EAA
QEAA
Selective Disclosure
Trust Model
Trusted List
Issuer
User
```

Reason:

These terms directly affect how the prototype should model verification templates, simulated wallet responses, privacy warnings, and audit logs.

## Simple project example

Example use case: age verification for an online service.

| Concept | Example in the use case |
|---|---|
| User | Person trying to access the service. |
| Relying Party | Online service that needs to check age. |
| Attribute | Age-over-18 claim. |
| PID | Full identity data, usually too much for this simple use case. |
| Selective Disclosure | User shares only proof of being over 18. |
| Issuer | Authority or provider that issued the underlying proof. |
| Trust Model | Rules used to decide whether the proof can be accepted. |
| Audit Log | Record that age-over-18 was checked, without storing unnecessary personal data. |

## Project boundary reminder

This project may simulate:

- requested attributes
- wallet responses
- issuer information
- trust-check outcomes
- validation results
- audit events

This project does not implement:

- a real EUDI Wallet
- real PID issuing
- real EAA/QEAA issuing
- real QTSP services
- real trusted-list validation
- legal certification
