Each glossary entry should help answer:

- What does the term mean?
- What is a simple example?
- Why does it matter for my project?
- How does the role or concept relate to the other ecosystem actors?

# Part 1 — ARF Definitions

## ARF Definitions — simple explanation and examples

| Term | Simple explanation | Simple example | Relevance to my project |
|---|---|---|---|
| Attribute | A single piece of information about a person, organization, or entity. | Date of birth, age over 18, address, driving licence category, company registration number. | Verification templates request attributes. The project must help decide which attributes are necessary and which are excessive. |
| Authentic Source | The original or legally recognized source of truth for an attribute. | A population register for legal name/address, a driving licence authority for driving privileges, a company register for company data. | The prototype should distinguish between user-provided data and data that comes from a trusted/authentic source. |
| Electronic Attestation of Attributes (EAA) | A digital proof that confirms certain attributes. | A digital proof that someone has a university degree, professional qualification, or residence status. | The project may simulate wallet responses that contain attribute attestations instead of full identity documents. |
| Issuer | The actor that gives PID or attribute attestations to the user/wallet. | A state authority issues PID; a university issues a degree attestation; a professional chamber issues a licence attestation. | Simulated wallet responses should show who supposedly issued the data, because issuer trust affects whether the relying party accepts it. |
| National Accreditation Body (NAB) | A national body that accredits organizations that assess conformity. | A national accreditation body authorizes a conformity assessment body to assess whether a wallet solution meets required standards. | Mostly background for version 1. It helps explain certification and trust governance, but the prototype does not perform accreditation. |
| Person Identification Data (PID) | Core identity data used to establish who a natural or legal person is. | Name, date of birth, personal identifier, nationality, or legal-person identity data depending on context. | PID is central when a business process needs identity proof. The prototype must avoid requesting full PID when a smaller proof is enough. |
| Person Identification Data Provider | A Member State or legal entity that provides PID to users. | A national identity authority provides identity data to a citizen's wallet. | In version 1 this is simulated, but the project should still model PID as coming from a responsible provider. |
| Public Key Infrastructure (PKI) | A technical trust system for managing public keys and checking that digital signatures belong to the right entity. | A relying party checks a credential signature using trusted public keys. | Full PKI validation is out of scope for version 1, but the audit log can record simulated trust checks. |
| Qualified Electronic Attestation of Attributes Provider | A qualified provider that issues high-trust electronic attestations of attributes. | A qualified provider issues an official professional qualification attestation. | Useful for future high-assurance scenarios. In version 1, QEAA can be represented as a higher-trust credential type. |
| Qualified Electronic Signature Creation Device (QSCD) | Approved software or hardware used to create qualified electronic signatures. | A secure signing device or certified signing app used to create a qualified electronic signature. | Mostly out of scope for the current verifier prototype unless future use cases include signing. |
| Qualified Trust Service Provider (QTSP) | A trust service provider that has qualified status from the supervisory body. | A qualified provider that issues qualified certificates or qualified attestations. | Important trust ecosystem context. The prototype should not claim to be a QTSP. |
| Relying Party | A person or organization that relies on electronic identification or trust services. In the wallet context, it relies on data presented from an EUDI Wallet. | A bank, employer, public office, online platform, or age-restricted service requesting wallet proof. | This is the main project perspective. The prototype supports the relying-party side by turning business needs into verification requests. |
| Selective Disclosure | The wallet capability that lets a user share only selected attributes instead of everything. | Sharing only proof of being over 18 instead of full date of birth and address. | Core privacy principle for the prototype. The system should warn when a request asks for more data than needed. |
| Trust | Willingness to rely on another actor to perform actions or make claims. | A relying party trusts that a PID provider issued correct identity data. | The prototype should make trust assumptions explicit, especially when simulated data is accepted. |
| Trust Framework | A legally enforceable set of operational and technical rules for a multi-party system. | Rules that define who may issue credentials, how they are checked, and what obligations participants have. | Useful for explaining that EUDI is not just technology; it depends on governance, rules, and responsibilities. |
| Trust Model | The rules that ensure ecosystem components and actors are legitimate. | Checking that a wallet, issuer, credential type, and relying party are recognized and valid. | Version 1 can simulate trust checks as business logic, while clearly stating that real trust validation is out of scope. |
| Trust Service Provider (TSP) | A person or organization that provides one or more trust services, qualified or non-qualified. | A provider offering electronic signatures, seals, timestamps, or attribute attestations. | Helps separate normal service providers from qualified providers. Mostly background unless a use case requires trust services. |
| Trust Service | A regulated electronic service connected to signatures, seals, timestamps, attestations, certificates, archiving, remote signing devices, or ledgers. | Electronic signature validation, timestamping, electronic attestation of attributes, or website authentication certificates. | Helps understand what kind of services exist around identity and trust. The prototype should not pretend to provide regulated trust services. |
| Trusted List | A list of authoritative entities and their current or historical trust status. | A list showing which providers are trusted or qualified in a legal context. | Real trusted-list integration is out of scope for version 1, but the concept is important for future validation logic. |
| User | The natural or legal person using an EUDI Wallet. | A citizen using a wallet to prove identity, or a company representative using a wallet for business authentication. | The prototype must respect that the user controls what is presented from the wallet. |
| EUDI Wallet Instance | A specific wallet installation or instance that belongs to and is controlled by a user. | The wallet app installed and activated on one person's phone. | Useful for understanding that the user interacts through a concrete wallet instance, not an abstract system. |
| EUDI Wallet Provider | The public or private organization responsible for operating an eIDAS-compliant wallet solution. | A Member State or approved organization providing a wallet app. | Out of scope for building, but important for ecosystem mapping. The project is not a wallet provider. |
| EUDI Wallet Solution | The complete wallet product and service offered by a wallet provider. | A certified wallet app plus supporting backend/service environment. | Helps define project boundaries: this prototype is not an EUDI Wallet Solution. |

# Part 2 — ARF 4.1 Ecosystem Roles: Application-Oriented View

## Why this section matters

The ecosystem role diagram is useful because it shows that the EUDI Wallet is not only an app. It is a network of actors.

For this project, the most important practical idea is:

```text
A business does not just request data from a user.
It relies on a chain of wallet, issuer, provider, schema, trust, and governance relationships.
```

The prototype should not implement all of these relationships in version 1, but it should represent them clearly enough for business process modeling, privacy checks, and audit logs.

---

## Ecosystem roles — real-life examples and project relevance

| No. | Ecosystem role | Simple explanation | Real-life example | Main relationship in the diagram | Relevance to my project |
|---:|---|---|---|---|---|
| 1 | End User of EUDI Wallet | The person or legal entity that owns/uses the wallet and decides what to present. | A citizen using a wallet app to prove they are over 18, or a company representative proving authority to act for a company. | User controls / activates the EUDI Wallet Instance. User receives services, support, wallet solution, and device from other actors. | Central for consent and selective disclosure. The prototype must model that data presentation is user-controlled. |
| 2 | EUDI Wallet Provider | The actor responsible for providing the wallet solution to users. | A Member State or approved provider offering the official wallet app. | Provides wallet solution and support to the user. Registers with the Trusted List Provider. May receive support/tools/libraries from device or subsystem providers. | Mostly out of scope. Important for explaining that this project is not a wallet provider. |
| 3 | PID Provider | The actor that provides Person Identification Data. | A national identity authority issuing identity data to a citizen's wallet. | Issues PID to the EUDI Wallet Instance. Registers with the Trusted List Provider. May receive qualified data from an Authentic Source. | High relevance. Simulated wallet responses may contain PID, and the prototype should know when PID is required or excessive. |
| 4 | Trusted List Provider | The actor that maintains trusted lists of recognized ecosystem actors or trust status. | A national or EU-level trust list service showing which providers are trusted/qualified. | Receives registrations from PID, EAA/QEAA, wallet providers, relying parties, and other relevant actors. | Medium-to-high relevance. Real validation is out of scope, but the prototype can simulate trusted-list checks and log the result. |
| 5 | QEAA Provider | The actor that issues Qualified Electronic Attestations of Attributes. | A qualified provider issuing an official professional licence, regulated qualification, or high-assurance attribute. | Issues QEAA to the EUDI Wallet Instance. Registers with the Trusted List Provider. Can use qualified data from Authentic Sources. | Medium relevance for version 1, high for future. Useful for high-trust verification use cases. |
| 6 | EAA Provider | The actor that issues non-qualified Electronic Attestations of Attributes. | A university issuing a degree attestation, or an organization issuing membership/status proof. | Issues EAA to the EUDI Wallet Instance. Registers with the Trusted List Provider. Can use data from Authentic Sources. | High relevance. Many business use cases need attributes, not full identity. |
| 7 | Qualified / non-qualified certificate provider for electronic signature or seal | Actor that provides certificates used for electronic signatures or seals. | A provider issuing a signing certificate to a person or an electronic seal certificate to a company. | Provides qualified certificates or related trust material to the EUDI Wallet Instance. Registers with the Trusted List Provider. | Low for version 1 unless the prototype later supports signing/sealing use cases. |
| 8 | Authentic Source | The authoritative source from which reliable data originates. | Population register, company register, driving licence register, professional register, education register. | Provides qualified or authoritative data to PID, EAA, and QEAA providers. | High conceptual relevance. Helps explain why some claims are more trustworthy than self-declared user data. |
| 9 | Relying Party | The organization or service that requests and relies on wallet-presented data. | Employer, bank, public service, online platform, car rental company, age-restricted service. | Receives presented PID/(Q)EAA from the EUDI Wallet Instance. Registers with the Trusted List Provider. Offers services to the user. | Highest relevance. The prototype represents the relying-party-side integration layer. |
| 10 | Conformity Assessment Body (CAB) | Actor that assesses whether wallet solutions or providers meet required standards. | An independent assessment organization checking whether a wallet implementation conforms to requirements. | Governance role. Connected to certification/conformity rather than the direct wallet presentation flow. | Low for version 1. Useful for explaining future certification context. |
| 11 | Supervisory Body | Authority that supervises certain actors and trust services. | A national authority supervising trust service providers or qualified providers. | Governance role outside the direct operational flow. | Low for version 1. Useful legal/governance context. |
| 12 | Device manufacturers and related subsystem providers | Actors that provide devices, operating systems, secure elements, libraries, or technical subsystems. | Smartphone manufacturer, mobile OS provider, secure element provider, wallet SDK/tooling provider. | Provides device to the user and support/tools/libraries to wallet-related actors. | Low-to-medium relevance. Helps explain that real wallets depend on secure device and OS infrastructure, which the prototype does not implement. |
| 13 | (Q)EAA Schema Provider | Actor that provides schemas defining the structure of EAA/QEAA. | A schema provider defining what fields a university degree attestation or professional licence attestation must contain. | Governance/standardization role. The diagram shows it as outside the direct ARF flow. | Medium relevance. Future verification templates should align with expected credential/attribute schemas. |
| 14 | National Accreditation Body | National body that accredits conformity assessment bodies. | A national accreditation authority authorizing a CAB to perform conformity assessments. | Governance role. Supports trust in CABs and certification processes. | Low for version 1. Useful only for explaining governance chain and certification background. |

## Relationship map from the ecosystem diagram

This section translates the picture into simple process relationships.

| Relationship | Plain meaning | Real-life example | Prototype interpretation |
|---|---|---|---|
| User controls / activates EUDI Wallet Instance | The user owns/uses the wallet instance and activates it for use. | A citizen opens the wallet app and approves sharing a proof. | Simulated wallet response should represent user-controlled consent/presentation. |
| PID Provider issues PID to Wallet Instance | Identity data is issued into the wallet. | A national identity authority issues name/date-of-birth data into the wallet. | Simulate PID as a credential/claim source. |
| QEAA Provider issues QEAA to Wallet Instance | A qualified provider issues high-trust attribute attestations. | A regulated professional licence is issued as a qualified attestation. | Future high-assurance scenario; version 1 can mark it as `credential_type = QEAA`. |
| EAA Provider issues EAA to Wallet Instance | A provider issues non-qualified attribute attestations. | A university issues a degree attestation. | Useful for common business examples such as education, membership, or status checks. |
| Certificate provider provides certificate to Wallet Instance | Signature/seal certificate material can be available through the wallet. | A user has a certificate for electronic signing. | Mostly out of scope unless signing is added later. |
| Wallet Instance presents PID/(Q)EAA to Relying Party | The wallet presents selected data or proofs to the organization requesting them. | User shares age-over-18 proof with an online service. | Core prototype flow: verification request → wallet response → validation result. |
| Relying Party offers services to User | The business or public service provides the service that requires verification. | A bank opens an account after identity verification. | Business process starts from a service need, not from technology. |
| EUDI Wallet Provider provides wallet solution to User | The user receives/uses a wallet solution from a provider. | A Member State wallet app is made available to citizens. | Out of scope; mention only to avoid confusion. |
| EUDI Wallet Provider provides support to User | The provider supports wallet users. | Helpdesk, account recovery, wallet lifecycle support. | Out of scope for prototype. |
| Device manufacturer provides device to User | The wallet runs on a device/OS environment. | Smartphone and operating system support secure wallet use. | Out of scope, but relevant for real-world deployment context. |
| Device/subsystem providers provide tools/libraries/support | Technical ecosystem actors support wallet providers and implementations. | OS APIs, secure enclave, SDKs, cryptographic libraries. | Out of scope for business prototype. |
| Authentic Source provides qualified/authoritative data | Providers rely on authoritative data sources. | A PID provider uses population register data. | Conceptually important when explaining data quality and trust. |
| Providers register with Trusted List Provider | Relevant providers are registered so others can know who is trusted/recognized. | A QEAA provider appears on a trusted list. | Simulate with `issuer_trusted = true/false` in version 1. |
| Relying Party registers with Trusted List Provider | Relying parties may need to be registered/recognized. | A service provider is listed as a legitimate relying party. | Future feature: check whether a relying party/request is allowed. |

## Example: age verification flow

| Step | Actor | What happens | Prototype equivalent |
|---:|---|---|---|
| 1 | Relying Party | Online service needs to check whether the user is over 18. | Create verification template: `age_over_18`. |
| 2 | User | User wants access to the service. | Simulated user starts verification flow. |
| 3 | Wallet Instance | Wallet presents only the needed proof. | Simulated wallet response contains `age_over_18 = true`. |
| 4 | PID Provider / EAA Provider | Underlying proof was issued by a recognized provider. | Simulated issuer metadata is attached. |
| 5 | Trusted List Provider | In real life, trust status may be checked. | Version 1 stores simulated result: `issuer_trusted = true`. |
| 6 | Relying Party | Service accepts or rejects the proof. | System validates response against template. |
| 7 | Audit Log | Verification result is recorded. | Store event without unnecessary personal data. |

Privacy lesson:

```text
The relying party should request proof of age-over-threshold, not full PID, unless full identity is truly required.
```

## Example: employment onboarding flow

| Step | Actor | What happens | Prototype equivalent |
|---:|---|---|---|
| 1 | Relying Party | Employer needs to verify identity and qualification. | Create onboarding verification template. |
| 2 | User | Candidate presents wallet proofs. | Simulated wallet response includes PID subset and qualification attestation. |
| 3 | PID Provider | Provides identity data. | Simulated PID issuer: national identity authority. |
| 4 | EAA/QEAA Provider | Provides qualification or professional status. | Simulated degree/licence attestation. |
| 5 | Authentic Source | Original data may come from register/university/professional body. | Metadata records source type. |
| 6 | Trusted List Provider | Provider trust status may be checked. | Simulated trust result. |
| 7 | Relying Party | Employer decides whether requirements are satisfied. | Validation result: accepted / rejected / missing claims. |
| 8 | Audit Log | The verification event is recorded. | Store purpose, requested claims, result, warnings. |

Privacy lesson:

```text
The employer should request only the identity and qualification attributes needed for onboarding, not unrelated wallet data.
```

## Example: residence verification flow

| Step | Actor | What happens | Prototype equivalent |
|---:|---|---|---|
| 1 | Relying Party | Service needs proof of residence or country. | Create residence verification template. |
| 2 | User | User agrees to present residence proof. | Simulated consent/presentation. |
| 3 | Authentic Source | Residence data originates from an official register. | Source metadata: population register. |
| 4 | PID Provider or EAA Provider | Issues identity/residence-related data. | Simulated credential type: PID or EAA depending on use case. |
| 5 | Wallet Instance | Presents selected residence attribute. | Wallet response contains required claim only. |
| 6 | Relying Party | Accepts/rejects proof. | Validation checks claim, issuer, freshness, and purpose. |
| 7 | Audit Log | Records the event. | Log result and privacy warning if request was excessive. |

Privacy lesson:

```text
If only country of residence is needed, requesting full address may be excessive.
```

## How this affects the prototype design

The ecosystem role model suggests these prototype objects:

| Prototype object | Connected ecosystem role | Purpose |
|---|---|---|
| Verification Template | Relying Party | Defines business purpose and required claims. |
| Requested Claim | Attribute / PID / EAA / QEAA | Defines what data is needed. |
| Simulated Wallet Response | User + EUDI Wallet Instance | Represents what the user presents. |
| Issuer Metadata | PID Provider / EAA Provider / QEAA Provider | Describes who issued the credential or attribute. |
| Source Metadata | Authentic Source | Describes where the data originally comes from. |
| Trust Check Result | Trusted List Provider / Trust Model | Simulates whether the issuer/provider is accepted. |
| Privacy Warning | Selective Disclosure / Data Minimization | Flags excessive requests. |
| Audit Log Event | Relying Party | Records what was requested, validated, rejected, or warned. |

## Boundary reminder

The project may simulate:

- requested attributes
- wallet responses
- issuer information
- source information
- trust-check outcomes
- validation results
- audit events
- privacy warnings

The project does not implement:

- a real EUDI Wallet
- a real wallet provider service
- real PID issuing
- real EAA/QEAA issuing
- real trust-list integration
- real QTSP services
- real device/OS security
- real conformity assessment
- legal certification
